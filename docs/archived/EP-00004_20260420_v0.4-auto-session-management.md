# EP-00004 — v0.4 Auto Session Management

## Problem / Pain Points
- Agents must manually call `session_start` before any session logging can happen — easy to forget, adds friction
- Claude Code hooks are tool-specific — other MCP clients (opencode, openclaw, hermes, codex) have no way to auto-start sessions
- If an agent forgets to start a session, turns are lost entirely — there's no fallback
- The current `session_log` requires a valid `session_id`, so any tool call before `session_start` fails silently or errors
- No way to continue a previous session across conversations — each conversation starts fresh

## Suggested Solution
Two-layer approach:

1. **MCP server auto-session** — the server tracks an active session per process and auto-creates one when a session-dependent tool is called without an active session. This works universally for all MCP clients with zero configuration.
2. **Claude Code hooks** — use `SessionStart` and `SessionEnd` hook events for explicit session lifecycle control. The `SessionStart` hook creates a session and persists the session ID via `$CLAUDE_ENV_FILE` so it's available throughout the conversation. The `SessionEnd` hook ends the session and surfaces the session ID for future continuation.

## Design Decisions

### Nullable project
`project_id` is **optional** for auto-created sessions. Sessions can exist with `project_id = NULL`, meaning "work is being logged but not yet assigned to a project."

- All session operations work without a project: `session_log`, `session_end`, reviews, maintenance
- `session_start` returns no `recommended_context` for NULL-project sessions (no tags to match against)
- If the agent or hook later provides a project (via explicit `session_start`), the NULL is **backfilled** on the existing session
- Reviews process NULL-project sessions normally — they appear as "unassigned" in stats

This avoids the need for default-project heuristics or config. The server never has to guess which project you're working on.

### Agent auto-detection
The agent name is derived from the **MCP client info** sent during the initialization handshake (`clientInfo.name`). MCP-compliant clients always send this — e.g. "claude-code", "opencode", etc.

- Auto-session uses `clientInfo.name` as the agent — no guessing needed
- Claude Code hooks can override with `--agent claude` for clarity
- Explicit `session_start` takes precedence over auto-detected value
- Fallback to "unknown" only for non-compliant clients that omit `clientInfo`

### Session ID ownership
Agent-knowledge generates its **own UUIDs** as session IDs. Claude Code does not expose its internal session ID to hooks (no `$CLAUDE_SESSION_ID` env var). Other clients may or may not have their own session concept.

To bridge the gap in Claude Code:
- `SessionStart` hook calls `akw session start` → receives the session ID
- Hook writes `export AKW_SESSION_ID=<id>` to `$CLAUDE_ENV_FILE` → persists it for all subsequent Bash commands in that conversation
- `SessionEnd` hook calls `akw session end` (no session ID flag) → finds and ends the most recent open session
- The session ID is displayed to the user on exit for future reference

### Auto-session vs hooks: mutual exclusion
Auto-session (Phase 1) and hooks (Phase 3) solve the same problem for different clients. When hooks are active (Claude Code), auto-session defers to the hook-created session to avoid duplicates.

The flow when **hooks are active** (Claude Code):
1. `SessionStart` hook fires → `akw session start` **closes any existing open sessions** (orphan cleanup) → creates new session in SQLite → writes `$AKW_SESSION_ID` to env
2. MCP server's `_ensure_session()` is called on first tool use → queries SQLite for the most recent open session → **finds the hook-created session** → adopts it as `_active_session`
3. No duplicate created. All turns go to the hook's session.

The flow when **no hooks** (opencode, codex, hermes, etc.):
1. No hook fires — no session exists
2. MCP server's `_ensure_session()` is called on first tool use → queries SQLite → no open session → **auto-creates one**
3. Agent can optionally call `session_start` later to upgrade the auto-session with metadata

This works because:
- `SessionStart` hook is synchronous — it completes before the conversation begins, so the session exists in SQLite before any MCP tool is called
- `_ensure_session()` always checks SQLite first, never creates blindly
- `_ensure_session()` picks the most recently created open session (`ORDER BY started_at DESC LIMIT 1`) to avoid adopting stale orphans

### Session continuation
Users can continue a previous session in a new conversation by passing its session ID. Continuation is handled **entirely by the agent via the MCP tool**, not by hooks — the hook always creates a fresh session on startup.

- `session_start` accepts an optional `continue_session` param
- If provided, the current session (if any) is **ended normally** (preserving any turns it may have) and the continued session is **reopened** (`ended_at` cleared)
- New turns append to the existing session
- `recommended_context` includes the session's previous turns for agent continuity
- CLI: `akw session start --continue <session_id>`
- UX: user sees session ID on exit, tells the agent next time ("continue session abc123")
- `akw session list --recent` shows recent sessions for easy lookup

### Explicit vs auto-session conflict
If `session_start` is called and an auto-session already exists for this process, **upgrade** the auto-session with the provided metadata (project_id, agent, type) rather than creating a second session. This avoids duplicate sessions when an agent bootstraps after the server already auto-created one.

### Idempotent session_end
`session_end` must be **idempotent** — calling it on an already-ended session is a no-op. This is necessary because both the agent (via MCP) and the `SessionEnd` hook (via CLI) may call `session_end` for the same session:

1. Agent follows wrapup prompt → calls `session_end` via MCP → session closed
2. User exits → `SessionEnd` hook fires → `akw session end` → session already closed → no-op

Implementation: `UPDATE sessions SET ended_at = ? WHERE id = ? AND ended_at IS NULL` — only updates if not already ended.

### Session wrapup and summarization
Summarization uses a **primary + fallback** strategy:

1. **Primary: agent summarizes before ending** — the `session_wrapup` MCP prompt guides the agent to summarize turns, write a session draft (`drafts/sessions/...`), and then call `session_end`. This produces the best quality summaries because the agent has full conversation context.
2. **Fallback: `akw review` batch job** — if the agent doesn't summarize (crash, user exits, forgot), the session remains in "ended but unreviewed" state. The `akw review` CLI command (run manually or via cron) batch-processes all pending sessions using the LLM to generate summaries from raw turns.

The `SessionEnd` hook in Claude Code calls `session_end` via CLI as a safety net — it ensures the session is always closed even if the agent didn't call `session_end` explicitly. Since `SessionEnd` fires when the conversation terminates (not per-turn), this correctly closes the session once.

For sessions that end without any summary (orphaned, crashed), the existing orphan auto-close + `akw review` pipeline handles cleanup.

## Implementation Phases

### Phase 1 — Server-Side Auto-Session
- [ ] Make `project_id` nullable in `create_session()` (storage layer + schema)
- [ ] Capture MCP `clientInfo.name` during server initialization and store as `_client_agent`
- [ ] Add `_active_session` module-level state in server.py (tracks current session ID per server process)
- [ ] Implement `_ensure_session()` helper:
  - If `_active_session` is set and still open, return it
  - Otherwise, query SQLite for the most recent open session (`ORDER BY started_at DESC LIMIT 1`) — if found, adopt it as `_active_session` and return it
  - Otherwise, create session with: project_id=NULL, agent=`_client_agent`, type="coding"
  - Set `_active_session` and return session info
- [ ] Update `session_start`:
  - If `continue_session` param provided: end the current session normally (preserving turns), reopen the continued session (clear `ended_at`), load previous turns, set as `_active_session`
  - Else if `_active_session` already exists (auto-created or hook-created): upgrade it with the caller's metadata (backfill project_id, agent, type) instead of creating a new session
  - Else: create new session, set as `_active_session`
- [ ] Update `session_log` to accept optional `session_id` — if omitted, use `_ensure_session()` to get/create active session
- [ ] Return `auto_session_created: true` flag in responses when a session was auto-created
- [ ] Make `session_end` idempotent — `UPDATE ... WHERE ended_at IS NULL`, no-op if already ended. Clear `_active_session`. Return the session ID in the response.

### Phase 2 — CLI Session Commands
- [ ] `akw session start [--project <id>] [--agent <name>] [--type <type>] [--continue <session_id>]`:
  - **Closes any existing open sessions** before creating a new one (orphan cleanup)
  - Creates or reopens session, prints session ID to stdout
- [ ] `akw session end [--session <id>]`:
  - If `--session` provided: end that specific session (idempotent)
  - If omitted: find the most recent open session and end it
  - Prints session ID of the ended session
- [ ] `akw session status` — shows active session info
- [ ] `akw session list [--recent] [--project <id>]` — lists recent sessions for continuation lookup
- [ ] These commands write directly to SQLite so they work independently of the MCP server process

### Phase 3 — Claude Code Hooks
- [ ] Create hook shell script `.claude/hooks/session-start.sh`:
  - Calls `akw session start --agent claude` (this closes orphans + creates new session)
  - Captures session ID from stdout
  - Writes `export AKW_SESSION_ID=<id>` to `$CLAUDE_ENV_FILE`
- [ ] Create hook shell script `.claude/hooks/session-end.sh`:
  - Calls `akw session end` (no `--session` flag — finds most recent open session)
  - Displays session ID for future continuation
- [ ] Configure hooks in `.claude/settings.json`:
  ```json

    "hooks": {
      "SessionStart": [{
        "hooks": [{
          "type": "command",
          "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/session-start.sh"
        }]
      }],
      "SessionEnd": [{
        "hooks": [{
          "type": "command",
          "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/session-end.sh"
        }]
      }]
    }
  }
  ```

## Design Note: Hook vs MCP Server Shared State

Claude Code hooks call CLI commands (`akw session start/end`) which write directly to SQLite. The MCP server also manages sessions via SQLite. Both share the same database, but the MCP server also has in-process state (`_active_session`).

Consistency is maintained because:
- `SessionStart` hook is synchronous — completes before any MCP tool call, so the session exists in SQLite when `_ensure_session()` first runs
- `akw session start` (CLI) closes any existing open sessions before creating — handles orphan cleanup from crashed conversations
- `_ensure_session()` queries SQLite for the most recent open session first, adopting hook-created sessions into `_active_session`
- `session_start` (MCP tool) checks if a session exists (hook-created or auto-created) and upgrades it
- `session_end` is idempotent — safe to call from both agent (MCP) and hook (CLI)
- `SessionEnd` hook uses `akw session end` (no session ID) — finds the most recent open session, correct even after session continuation
